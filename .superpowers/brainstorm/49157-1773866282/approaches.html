Program Carrier Tracking — Architecture Approaches

How should the new program_carriers table integrate with the existing system?

A

Standalone Table + Reconciler Enhancement

New table: program_carriers with columns: id, program_id (FK→policies.id), carrier, policy_number, premium, limit_amount, sort_order

Reconciler change: When matching imported rows against programs, query program_carriers for exact carrier + policy number matches (+30 bonus) instead of substring matching program_carriers text (+15)

Batch create: POST /reconcile/batch-create-program creates program record + inserts one program_carriers row per selected import row

UI: Contenteditable matrix on policy edit page replaces the textarea. Client detail _programs.html shows nested carrier rows.

Migration path: Migration parses existing comma-separated program_carriers text into rows in the new table

Pros

  • Clean separation — program_carriers is its own table
  • Reconciler gets structured data for precise matching
  • Backward compatible — existing programs migrated automatically
  • Simple schema, easy to query

Cons

  • Keeps the text field program_carriers on policies table (redundant until removed)
  • Two sources of truth during transition
B

Standalone Table + Deprecate Text Field

Same as A, but also:

Deprecation: Stop writing to program_carriers text field. Keep the column but ignore it. All reads come from the new table. program_carrier_count becomes a COUNT() on the table.

Views updated: v_policy_status, v_schedule join to program_carriers table for carrier display instead of reading the text field

Reconciler: Drops the +15 text substring bonus entirely. Uses only structured table lookups with graduated bonuses (+30 exact policy number, +10 carrier name match)

Pros

  • Single source of truth from day one
  • No redundant data to keep in sync
  • Views and queries are cleaner
  • program_carrier_count is always accurate (computed, not stored)

Cons

  • More migration work — must update views, reconciler, all read paths
  • Slightly larger blast radius
C

JSON Column Instead of Table

No new table. Replace program_carriers text with a JSON array: [{"carrier":"AIG","policy_number":"POL-123","premium":50000,"limit":1000000}, ...]

Reconciler: Parses JSON, matches against carrier + policy_number fields

UI: Same contenteditable matrix, but backed by JSON read/write instead of table CRUD

Pros

  • No new table or migration complexity
  • Atomic — program and its carriers saved together

Cons

  • Can't JOIN or query carrier rows independently
  • SQLite JSON functions are clunky for matching
  • Harder to do per-carrier diffs in reconcile
  • Breaks the pattern of relational data throughout the app

Recommendation: B — The text field was always a stopgap. A proper table with full deprecation of the text field gives you one source of truth, clean queries, and a reconciler that can do real structured matching. The extra migration work is modest since you're already touching these code paths.
